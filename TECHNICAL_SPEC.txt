4. DATABASE SCHEMA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: episodes.db  (SQLite, single file, located in server/ working directory)
Driver: aiosqlite — all operations are async; connections are opened and closed
        per-operation (no connection pool — SQLite serialises writes natively).

TABLE: episodes
  id               INTEGER  PRIMARY KEY AUTOINCREMENT
  start_time       TEXT     NOT NULL   ISO-8601 datetime string (UTC)
  end_time         TEXT     NOT NULL   ISO-8601 datetime string (UTC)
  peak_stress      REAL     NOT NULL   0.0 – 100.0
  avg_bpm          REAL     NOT NULL   beats per minute
  duration_seconds REAL     NOT NULL   (end_time - start_time).total_seconds()
  analysis_json    TEXT     NULL       JSON string — see AI response schema below
  parent_notes     TEXT     NULL       Free-text parent observations

  Migration: parent_notes was added after initial schema via:
    ALTER TABLE episodes ADD COLUMN parent_notes TEXT
    Wrapped in try/except — SQLite has no IF NOT EXISTS for ALTER TABLE.
    Safe to run multiple times; the exception is swallowed on duplicate.

TABLE: episode_datapoints
  id         INTEGER  PRIMARY KEY AUTOINCREMENT
  episode_id INTEGER  NOT NULL   Foreign key → episodes.id (not enforced by SQLite)
  timestamp  TEXT     NOT NULL   ISO-8601 datetime string (UTC)
  bpm        REAL     NOT NULL
  stress     REAL     NOT NULL   0.0 – 100.0
  rmssd      REAL     NOT NULL

  Datapoints are bulk-inserted after each episode closes using executemany().
  There is no CASCADE delete — orphaned datapoints persist if an episode is
  manually deleted from the DB.

TABLE: telegram_subscribers
  chat_id  INTEGER  PRIMARY KEY

  INSERT OR IGNORE prevents duplicates when a user sends /start multiple times.
  No unsubscribe mechanism currently implemented.

QUERY PATTERNS
  get_episodes():        SELECT * FROM episodes ORDER BY start_time DESC LIMIT 20
  get_episode_datapoints(id): SELECT * FROM episode_datapoints WHERE episode_id = ?
                              ORDER BY timestamp
  get_summary():         SELECT * FROM episodes  (full table scan; acceptable at
                         current scale of ~hundreds of rows)
  get_heatmap():         SELECT start_time FROM episodes
  generate_report():     SELECT * FROM episodes WHERE start_time >= ? ORDER BY start_time

5. STRESS CALCULATION ALGORITHM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: server/stress_engine.py
Function: calculate_stress(bpm: float, rmssd: float) -> float

CALIBRATION CONSTANTS
  BPM_REST  =  68.0   Resting heart rate (calm baseline)
  BPM_MAX   =  96.0   Upper heart rate reference for normalization
  RMSSD_REST = 45.0   Resting RMSSD (calm baseline — high HRV = low stress)
  RMSSD_MIN  = 15.0   Minimum expected RMSSD under high stress

RMSSD SCORE  (primary signal, weight 0.7)
  rmssd_score = (RMSSD_REST - rmssd) / (RMSSD_REST - RMSSD_MIN)
              = (45 - rmssd) / (45 - 15)
              = (45 - rmssd) / 30
  Clamped to [0.0, 1.0].
  When rmssd = 45 (rest): score = 0.0  (no stress)
  When rmssd = 15 (crisis): score = 1.0  (maximum stress)
  When rmssd < 15: clamped to 1.0

BPM SCORE  (secondary signal, weight 0.3)
  bpm_score = (bpm - BPM_REST) / (BPM_MAX - BPM_REST)
            = (bpm - 68) / (96 - 68)
            = (bpm - 68) / 28
  Clamped to [0.0, 1.0].
  When bpm = 68 (rest): score = 0.0
  When bpm = 96 (elevated): score = 1.0
  When bpm > 96: clamped to 1.0

COMBINED STRESS
  stress = (0.7 × rmssd_score + 0.3 × bpm_score) × 100
  Final value clamped to [0.0, 100.0] and rounded to 1 decimal place.

DESIGN RATIONALE
  RMSSD is weighted 70% because it is a direct measure of autonomic nervous
  system state. Heart rate elevation is a lagging indicator that also rises
  during exercise and excitement. RMSSD specifically drops under psychological
  and sensory stress even when heart rate has not yet changed significantly,
  giving earlier detection of the onset of a stress episode.

THRESHOLDS
  >= 70.0  Telegram warning alert (with 5-minute cooldown)
  >= 90.0  Telegram crisis alert + DB episode logging begins
  <  40.0  Episode resolution (if a Telegram episode was active)

6. BROADCAST LOOP AND EPISODE STATE MACHINE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: server/main.py
Function: broadcast_loop() — runs as an asyncio Task for the lifetime of the app.
Tick rate: 1 second (asyncio.sleep(1) at end of each iteration).

MODULE-LEVEL STATE (two parallel episode trackers)

  DB EPISODE TRACKER (threshold: stress >= 90.0)
    _episode_active:       bool     Is a crisis episode currently open?
    _episode_start:        datetime When the current episode started (UTC)
    _episode_peak_stress:  float    Highest stress seen in current episode
    _episode_bpm_readings: list     All BPM values seen during episode (for avg)
    _episode_datapoints:   list     Dicts {ts, bpm, stress, rmssd} per tick

  TELEGRAM EPISODE TRACKER (threshold: stress >= 70.0)
    _tg_episode_active:    bool     Is a warning-level episode open?
    _tg_episode_start:     datetime (uses local time, not UTC)
    _tg_peak_stress:       float
    _tg_bpm_readings:      list

  LAST KNOWN READINGS (for /nonverbal endpoint)
    _last_stress:          float
    _last_bpm:             float

LOOP LOGIC PER TICK

  1. Read from data source: raw = ds.read()
     Returns dict: {bpm, rmssd, rr_intervals, source}

  2. Calculate stress: stress = calculate_stress(raw["bpm"], raw["rmssd"])

  3. Update _last_stress, _last_bpm.

  4. Build JSON payload for WebSocket broadcast:
     {bpm, stress, rmssd, rr_intervals, source, alert}
     alert = (stress >= 90.0)  — boolean

  5. DB EPISODE STATE MACHINE:
     if stress >= 90:
       if NOT _episode_active:
         → Open new episode: set _episode_active=True, record _episode_start,
           reset _episode_peak_stress, _episode_bpm_readings, _episode_datapoints
       else:
         → Update: max peak, append BPM
       → Always: append datapoint to _episode_datapoints
     elif _episode_active:  (stress dropped below 90, episode just ended)
       → Close episode: set _episode_active=False
       → Snapshot datapoints list (copy before clearing)
       → asyncio.create_task(_finish_episode(...))  — fire-and-forget
         _finish_episode():
           1. await log_episode(start, end, peak, avg_bpm) → episode_id
           2. await analyze_episode(peak, avg_bpm, duration, time, day) → analysis dict
           3. await update_episode_analysis(episode_id, json.dumps(analysis))
           4. await log_datapoints(episode_id, captured_datapoints)

  6. TELEGRAM EPISODE STATE MACHINE:
     if stress >= 70 and NOT _tg_episode_active:
       → Open Telegram episode: record start, peak, bpm list
     elif _tg_episode_active:
       → Update: max peak, append BPM

     if stress >= 70:
       → asyncio.create_task(notifier.send_alert("warning", {...}))
     if stress >= 90:
       → asyncio.create_task(notifier.send_alert("crisis", {...}))

     if stress < 40 and _tg_episode_active:
       → Close Telegram episode
       → Calculate duration in minutes
       → asyncio.create_task(_notify_resolved(peak, avg_bpm, duration_min))
         _notify_resolved():
           1. await analyze_episode(...)  — get AI recommendation
           2. await notifier.send_alert("resolved", {...})
       → notifier.reset_episode()  — allows next crisis alert to fire

  7. WEBSOCKET BROADCAST:
     for ws in list(connected_clients):
       await ws.send_text(payload)
     Dead connections (any exception) are collected and removed.

  8. await asyncio.sleep(1)

NOTE ON CONCURRENCY:
  asyncio.create_task() is used for all I/O-bound operations that should not
  block the broadcast loop tick (DB writes, AI calls, Telegram sends). The
  loop never awaits these tasks directly. Exceptions inside tasks are silently
  dropped — the outer loop has a bare except: pass as final safety net.

7. REST API — ALL ENDPOINTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All endpoints use JSON. CORS is open (*) — any origin, any method, any headers,
credentials allowed. Defined in FastAPI using CORSMiddleware.

WebSocket
  WS /ws
    Client connects, server adds to connected_clients list.
    Server sends JSON text frames every ~1 second:
      { "bpm": float, "stress": float, "rmssd": float,
        "rr_intervals": [float, ...], "source": string, "alert": boolean }
    Client sends: anything (ignored — server only uses receive_text() to detect disconnect).
    On disconnect: WebSocketDisconnect exception → removed from list.

Episodes
  GET /episodes
    Response: array of episode objects (max 20, sorted start_time DESC)
    Episode object:
      { "id": int, "start_time": str, "end_time": str, "peak_stress": float,
        "avg_bpm": float, "duration_seconds": float,
        "analysis_json": str|null, "parent_notes": str|null }

  PATCH /episodes/{episode_id}/notes
    Body: { "notes": string }  (Pydantic NotesBody model)
    Action: UPDATE episodes SET parent_notes = ? WHERE id = ?
    Response: { "ok": true }

  GET /episodes/{episode_id}/datapoints
    Response: array of datapoint objects, sorted by timestamp ASC
    Datapoint object:
      { "id": int, "episode_id": int, "timestamp": str,
        "bpm": float, "stress": float, "rmssd": float }

Analytics
  GET /analytics/summary
    Response: AnalyticsSummary object
      { "total_episodes": int,
        "episodes_this_week": int,
        "episodes_this_month": int,
        "avg_peak_stress": float,
        "avg_duration_seconds": float,
        "risk_distribution": { "low": int, "medium": int, "high": int },
        "trend": "improving"|"stable"|"worsening",
        "top_triggers": [ { "trigger": str, "count": int }, ... ]  (max 5) }

  GET /analytics/heatmap
    Response: array of HeatmapCell objects
      [ { "day": int(0-6, Mon=0), "hour": int(0-23), "count": int }, ... ]
    Only cells with count > 0 are returned.

Reports
  GET /report/weekly
    Generates AI report covering last 7 days.
    Response: MedicalReport object
      { "period": "weekly", "generated_at": str,
        "summary": str, "findings": [str, ...],
        "recommendations": [str, ...], "risk_assessment": str }

  GET /report/monthly
    Same as above but covers last 30 days. period = "monthly".

Demo — Stress Simulator
  POST /demo/stress/{level}
    Path param: level float (0.0 – 1.0, where 1.0 = 100% stress)
    Action: calls data_source.set_stress(level)
    Response: { "ok": true }

  POST /demo/scenario/rising
    Action: calls await data_source.start_rising()
    Runs an async scenario: stress rises gradually from 0 to 0.95 over ~30 ticks.
    Response: { "ok": true }

  POST /demo/scenario/reset
    Action: calls data_source.reset()
    Response: { "ok": true }

Demo — Data Mode
  POST /demo/seed
    Action: calls await seed_demo_data()
    Inserts ~35 synthetic episodes with datapoints across the past 30 days.
    Response: { "ok": true, "episodes_created": int }

  POST /demo/clear
    Action: calls await clear_all_data()
    Deletes all rows from episode_datapoints, episodes, resets sqlite_sequence.
    Response: { "ok": true }

Nonverbal
  POST /nonverbal/{button}
    Path param: button string — one of "quiet", "home", "help"
    Action: reads _last_stress and _last_bpm from module globals,
            calls await notifier.send_alert("nonverbal", {...})
    Response: { "ok": true }

App Lifespan
  On startup: await init_db(), start broadcast_loop task, start telegram_poll_loop task.
  On shutdown: cancel both tasks.

8. AI INTEGRATION — EPISODE ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: server/ai_client.py
Client: AsyncOpenAI (singleton module-level instance)
Model: gpt-4o

SYSTEM PROMPT (verbatim):
  "You analyze sensory overload episodes for neurodiverse children.
   Return ONLY valid JSON with exactly these three fields:
   trigger (string — likely cause),
   recommendation (string — what the caregiver should do next),
   risk_level (exactly one of: low, medium, high).
   No markdown. No explanation. Nothing else."

USER MESSAGE FORMAT:
  "Peak stress: {peak_stress:.0f}%.
   Average BPM: {avg_bpm:.0f}.
   Duration: {duration_seconds:.0f} seconds.
   Time of day: {time_of_day}.   ← format: "HH:MM"
   Day of week: {day_of_week}."  ← format: "Monday", "Tuesday", etc.

RESPONSE SCHEMA (what the model returns):
  { "trigger": string, "recommendation": string, "risk_level": "low"|"medium"|"high" }

FALLBACK (on any exception — network, parse error, API error):
  { "trigger": "Unknown",
    "recommendation": "Monitor closely and remove the child from overstimulating environments.",
    "risk_level": "medium" }

CALLED FROM:
  1. _finish_episode() — immediately after each DB episode closes (via asyncio.create_task)
  2. _notify_resolved() — to get a recommendation text to include in the resolved Telegram message

9. AI INTEGRATION — MEDICAL REPORT GENERATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: server/report_generator.py
Client: AsyncOpenAI (separate module-level instance)
Model: gpt-4o

SYSTEM PROMPT (verbatim):
  "You are a pediatric neurologist assistant generating medical reports for neurodiverse children.
   Each episode entry includes biometric data (stress, BPM, duration) and may also include
   parent-observed context — what the caregiver personally noticed around the time of the episode
   (environment, sounds, activity, food, sleep, mood, etc.).
   Parent observations are first-hand qualitative data and should be weighted heavily alongside
   the biometric measurements. When parent notes are present, explicitly reference them in your
   findings and tailor recommendations to the specific triggers the parent identified.
   Return ONLY valid JSON with exactly these fields:

Real sensor mode (DATA_SOURCE=real):

  The read() interface is identical so the rest of main.py is unaffected.

WEBSOCKET CONNECTION:
  Exponential backoff: [1s, 2s, 4s, 8s, 16s, 30s] per retry.

  without requiring a page refresh.
  GET /episodes → setEpisodes(data)

CLEANUP:
  On component unmount: closes WebSocket, clears backoff timer, clears interval.

18. HARDWARE — ESP32 + MAX30102
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MICROCONTROLLER: ESP32 (any variant with WiFi)
SENSOR: MAX30102 — optical pulse oximeter and heart rate sensor
  Interface: I2C (SDA, SCL pins)
  Measurement: Red (660 nm) and infrared (880 nm) LEDs + photodetector
  Output: raw ADC values → firmware extracts BPM and RR intervals

WHAT THE FIRMWARE SENDS TO THE SERVER
  Per measurement cycle (target: every 1 second), the ESP32 POSTs or sends
  over WebSocket to the server:
  { "bpm": float, "rmssd": float, "rr_intervals": [float, ...], "source": "esp32" }

  RR intervals are the inter-beat intervals in milliseconds. The server's
  stress_engine does not compute RMSSD from raw RR intervals — it expects
  RMSSD to be pre-computed by the ESP32 firmware. The firmware uses a rolling
  window of the last N RR intervals.

  RMSSD formula (computed on device):
    RMSSD = sqrt( mean( (RR[i+1] - RR[i])^2 for i in range(N-1) ) )

SWITCHING FROM SIMULATOR TO REAL DEVICE
  Set DATA_SOURCE=real in server/.env (or Railway Variables).
  The data_source.py module checks this at startup and returns a RealSensor
  instance from get_data_source() instead of the simulator.
  Everything else in main.py is unchanged — same stress engine, same episode
  detection, same WebSocket broadcast.

COST BREAKDOWN
  ESP32 development board: ~$5–8 USD
  MAX30102 breakout module: ~$3–6 USD
  Total hardware per unit:  ~$8–14 USD

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BACKEND — RAILWAY
  Start command: python server/main.py
    (uvicorn.run("main:app", host="0.0.0.0", port=PORT) inside __main__ block)
  Environment variables set in Railway dashboard:
    OPENAI_API_KEY, TELEGRAM_BOT_TOKEN, TELEGRAM_USER_NAME
  PORT is injected automatically by Railway.
  SQLite DB file (episodes.db) lives in the working directory of the Railway
  container. It persists as long as the deployment is not restarted from zero.
  For production data persistence, a Railway Volume should be mounted.

FRONTEND — VERCEL
  Build command: npm run build  (runs tsc && vite build)
  Output directory: dist/
  Environment variables set in Vercel project settings:
    VITE_API_URL  (HTTPS URL of Railway backend)
    VITE_WS_URL   (WSS URL of Railway backend)
  These are inlined into the JS bundle at build time.

CORS
  Backend allows all origins (*), all methods, all headers, credentials=True.
  This is intentional for a hackathon prototype. Production would restrict
  allow_origins to the specific Vercel deployment URL.

KNOWN RECHARTS BUG WORKAROUNDS (implemented in DoctorDashboard.tsx)
  Bug: radius={[topLeft, topRight, bottomRight, bottomLeft]} on BarChart bars
       causes NaN SVG path coordinates when bar height is zero, crashing the
       entire React tree to a white screen.
  Fix 1: Only render BarChart when data has at least one bar with count > 0.
  Fix 2: Use radius={4} (scalar) instead of the array form.
  Fix 3: Set explicit YAxis domain={[0, maxCount + 1]} to prevent auto-scaling
         issues on small datasets.

20. SECURITY NOTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  - server/.env is in .gitignore and has never been committed.
  - OPENAI_API_KEY and TELEGRAM_BOT_TOKEN exist only in server/.env locally
    and in Railway Variables in production.
  - No authentication or authorization is implemented on any endpoint.
    This is intentional for a hackathon prototype. All endpoints are public.
  - The demo seed and clear endpoints are equally public. In production these
    would be protected by an admin token or removed entirely.
  - load_dotenv() is anchored to the server directory:
      load_dotenv(Path(__file__).parent / ".env")
    This prevents the CWD-relative path from failing when the server is
    started from a parent directory (which caused a Telegram bot token
    not-found bug during development).
  - SQLite queries use parameterized statements throughout (no string
    interpolation into SQL) — no SQL injection surface.
